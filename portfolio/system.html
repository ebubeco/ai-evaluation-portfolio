Evaluation System

The 8-step
evaluation loop

Every AI output evaluated through the same reproducible system. No impressionism. No anchor bias. No rubric theatre. Decision visible in the first sentence of every output.

Methodology

Step-by-step
evaluation protocol

01
Anchor
Always Run
Rule → If the anchor is wrong, everything downstream is wrong.
Strip the prompt to what must be true. Extract the core requirement, constraints, format, exclusions, and failure boundary before evaluating anything.

This is not a skim. Missing a constraint at this stage means the entire evaluation is built on a wrong foundation.
Core requirementExplicit constraintsFormat requirementsExclusionsFailure boundary
02
Instruction Check
Always Run
Rule → Clear instruction miss = failure. No rehabilitation.
Did the response do exactly that? Scan for missed constraints, wrong format, added assumptions, violated exclusions.

A response that executes every keyword while violating the instruction's intent is still a failure. The test is not "did it try?" — it's "did it do the thing?"
Missed constraintsWrong formatAdded assumptionsViolated exclusionsIntent vs literal
03
Reality Check
ConditionalTrigger: precise claims
Rule → One confirmed hallucination overrides everything else.
Test every factual claim: entailed by the task = valid. Merely compatible with the task = hallucination flag.

Confident framing is not evidence. Suspicious specificity always triggers this step.
False precisionPlausible but ungroundedConfident framingNamed entity claimsNumeric distortion
04
Logic Check
ConditionalAfter Step 3
Rule → Correct answer + broken reasoning = reasoning failure.
Evaluate reasoning only on facts that pass Step 3. A correct answer produced by broken reasoning is a reasoning failure. A wrong answer from valid reasoning is a factual failure.

If Step 3 fails, Step 4 only documents downstream damage.
Premise validityInference chainConclusion alignmentStep 3 dependency
05
Ambiguity Check
ConditionalTrigger: conflicting constraints
Rule → Silent assumption = failure. Explicit assumption = acceptable minimum.
Would another annotator read this the same way? Classify ambiguity as prompt-level or interpretation-level.

No ambiguity → don't invent it. Silent assumption → failure. Explicit assumption → acceptable minimum.
Prompt-level ambiguityInterpretation-levelSilent assumptionsConflicting constraintsDefensible interpretation
06
Root Cause
ConditionalTrigger: multiple failures
Rule → The root cause is the earliest failure in the loop. One fix.
When multiple failures are present, the root cause is the earliest failure in the evaluation loop.

Do not list symptoms. Name the cause. If your root cause cannot be expressed as a single fix, it is not the root cause — keep tracing upstream.
First failure pointCascade attributionSingle-fix testPrimary vs secondary cause
07
The Call
Always Run
Rule → Partial fulfillment of core requirement = task failure.
Make a single, decisive judgment. Classify internally: reliable / usable with weakness / unreliable.

Do not expose the classification in output. Partial fulfillment of the core requirement is task failure unless the remaining gap doesn't affect usability.

Epistemic and semantic uncertainty are valid. Normative disagreement is not.
ReliableUsable with weaknessUnreliableUsability gap test
08
Write It
Always Run
Rule → Decision visible in lead sentence. No rubric language.
One paragraph. Decision visible in the lead sentence. Include what the response did, the root issue if any, and why it matters.

1–2 core judgments only. No cataloging. No rubric language. No visible scoring structure. The decision must be instantly verifiable from the first sentence.
Decision-firstRoot issue if anyWhy it mattersNo structure exposed
Core Principles

What the system
is built on

P-01
Decision First
The most important judgment appears in the first sentence. Everything after is support. Readers should never have to hunt for the verdict.
Burying the decision is itself a quality failure.
P-02
Constraint Hierarchy Is Not Negotiable
Explicit constraints evaluated before response quality. A fluent, helpful response that violates an explicit exclusion is rejected. Fluency does not rehabilitate a constraint violation.
Operator-layer constraints > user-layer preferences.
P-03
One Confirmed Hallucination Overrides Everything
A single verified fabrication — regardless of the quality of the rest — is an automatic reject. Confident framing is not evidence. Suspicious specificity triggers explicit verification.
Entailed by the task ≠ merely compatible with it.
P-04
Root Cause, Not Symptom Catalogue
When multiple failures are present, the root cause is the earliest failure in the loop. Downstream failures are consequences, not co-equal failures. One fix traces to one cause.
If it needs multiple fixes, it's not the root cause.
P-05
Path Evaluated, Not Just Destination
A correct answer produced by invalid reasoning is a reasoning failure. A wrong answer from valid reasoning is a factual failure. The distinction matters for downstream model training.
Correct output + broken reasoning = reasoning failure.
P-06
Downstream Impact Is the Standard
Evaluation decisions are made with their downstream effect in mind. A small error that corrupts RLHF training signal is not a small error. The training impact is part of the severity assessment.
Assume downstream impact. Small errors are real errors.
Failure Taxonomy

Documented
failure classes

30+ recurring failure patterns documented across 1,200+ evaluation tasks.

Failure Class
What It Looks Like
Detection Method
Severity
Explicit Constraint Override
Operator-layer rule clearly stated in system context — user-turn response violates it. Often fluent and otherwise high-quality.
Step 1 anchor captures constraint. Step 2 checks for it before quality evaluation begins.
Hard Fail
Entailment-Gap Hallucination
Claim contextually plausible, specific, and confidently stated — but not grounded in prompt. Classic "confident fabrication."
Step 3: is this claim entailed by the task? Or merely compatible with it?
Hard Fail
Premise-Level Reasoning Error
Reasoning chain begins from a flawed or unverified premise. Conclusion may appear correct. The error is upstream.
Step 4 only after Step 3 passes. Evaluate the inference chain independently of the destination.
Hard Fail
Silent Assumption Injection
Genuine ambiguity in prompt — model resolves it by choosing an interpretation without flagging it. The assumption is never stated.
Step 5: would another annotator read this the same way? If no, was the assumption stated?
Hard Fail
Instruction-Intent Gap
All keywords technically executed. Core intent of the instruction violated. Passes a keyword-scan; fails an intent-scan.
Step 2: literal vs intent evaluation. "Did it do the thing?" not "did it try?"
Hard Fail
Visual Grounding Failure (Multimodal)
Fluent, coherent image description that misrepresents entities, spatial relationships, or attributes. Invisible to text-only evaluation.
Dual-layer protocol: text coherence evaluated independently from visual grounding accuracy.
Hard Fail
Training Data Distribution Bias
Systematic quality gap between knowledge domains — Western domains scoring significantly higher than matched non-Western prompts.
Parallel prompt battery: matched prompts across regions with identical rubrics. Track quality variance by domain.
Systemic Flag
Schema Drift (Data Ops)
Cross-batch labelling inconsistency with no hard errors. Passes per-entry quality checks. Fails cross-batch consistency audit.
Cross-batch consistency mapping. Per-entry accuracy is insufficient — cross-batch standard required.
Silent Corruption
Weak Reasoning (Recoverable)
Reasoning path is weaker than it needs to be but conclusion is still defensible. Not a reasoning failure — a confidence qualifier.
Step 4: is the conclusion still defensible given the reasoning quality? Affects confidence rating only.
